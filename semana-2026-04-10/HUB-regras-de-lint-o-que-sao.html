O que sao regras de lint e por que existem

Origem do termo lint

O nome "lint" vem de uma ferramenta criada em 1978 por Stephen C. Johnson nos laboratorios Bell. Ela analisava codigo C procurando construcoes suspeitas que o compilador aceitava mas que provavelmente eram bugs. O nome faz referencia aquelas bolinhas de tecido que grudam na roupa -- pequenos residuos que voce nao percebe ate olhar de perto.

Quatro decadas depois, o conceito evoluiu bastante. Um linter hoje cobre muito mais que erros de sintaxe. Ele verifica estilo de codigo, padroes de seguranca, acessibilidade em HTML, complexidade ciclomatica (quantidade de caminhos independentes que o codigo pode seguir) e ate importacoes desnecessarias. Cada regra e uma instrucao especifica que o linter aplica ao ler seu codigo fonte.

Como um linter funciona na pratica

Um linter opera em tres etapas. Primeiro, ele faz o parsing (leitura e interpretacao estrutural) do seu codigo, transformando texto em uma arvore sintatica abstrata (AST). Depois, percorre essa arvore aplicando cada regra configurada. Por fim, gera um relatorio com os problemas encontrados, separados entre warnings e errors.

Exemplo concreto. Se voce tem a regra no-unused-vars ativa no ESLint e escreve:

const apiKey = process.env.API_KEY;
const baseUrl = "https://api.hubdodesenvolvedor.com.br";
const timeout = 5000;

fetch(baseUrl + "/cpf", { headers: { Authorization: apiKey } });

O linter vai apontar que timeout foi declarado mas nunca usado. Parece bobo, mas variaveis orfas acumulam e poluem o codigo com o tempo.

Diferenca entre lint, formatacao e testes

Muita gente confunde linting com formatacao. Sao coisas diferentes. Formatadores como o Prettier cuidam da aparencia -- espacos, quebras de linha, aspas simples ou duplas. Ja o linter analisa a semantica e logica do codigo.

Testes verificam se o codigo funciona corretamente para inputs especificos. O linter verifica se o codigo segue padroes antes mesmo de rodar. Sao camadas complementares: lint pega problemas estruturais, testes pegam problemas de comportamento.

💡 DICA: Use formatador E linter juntos. O Prettier cuida do estilo visual, o ESLint cuida da qualidade logica. Configure o ESLint para nao reclamar de regras que o Prettier ja resolve -- use o plugin eslint-config-prettier para isso.

Principais ferramentas de lint por linguagem

Ferramentas para JavaScript e TypeScript

O ESLint domina o ecossistema JavaScript. Nicholas Zakas criou a ferramenta em 2013, e hoje ela passa de 40 milhoes de downloads semanais no npm. O ESLint e totalmente plugavel -- voce cria regras customizadas, usa presets da comunidade e integra com qualquer editor.

Para TypeScript, o typescript-eslint estende o ESLint com regras que entendem tipos. Ele detecta coisas como usar any quando existe um tipo especifico, ou promises nao tratadas.

Outra ferramenta ganhando tracao e o Biome (antigo Rome). Ele combina linting e formatacao num unico binario escrito em Rust. A execucao e muito rapida -- ate 35x mais veloz que ESLint em benchmarks com projetos grandes.

Ferramentas para Python

O Pylint e o linter classico do Python. Rigoroso por padrao, ele verifica desde nomes de variaveis ate complexidade de funcoes. Muitos devs acham ele verboso demais e migram para o Flake8, que e mais leve e combina tres checadores: PyFlakes, pycodestyle e McCabe.

O Ruff apareceu em 2022 e sacudiu o ecossistema. Escrito em Rust, ele reimplementa centenas de regras do Flake8 e outros linters Python rodando 10-100x mais rapido. Se voce trabalha com projetos Python grandes, Ruff e a melhor opcao disponivel hoje.

Cada linguagem tem seu ecossistema. Nao existe projeto profissional serio que nao use algum tipo de linting. E tao basico quanto usar controle de versao.

⚠️ ATENCAO: Nao tente ativar todas as verificacoes de uma vez em projetos legados. Voce vai gerar centenas de erros e desmoralizar o time. Use uma abordagem incremental: comece com regras de erro (bugs reais) e va adicionando regras de estilo gradualmente.

Como configurar regras de lint no seu projeto

Configuracao basica do ESLint

A configuracao do ESLint mudou com a versao 9, que adotou o formato flat config. Crie um arquivo eslint.config.js na raiz do projeto:

import js from "@eslint/js";

export default [
  js.configs.recommended,
  {
    rules: {
      "no-unused-vars": "warn",
      "no-console": "error",
      "eqeqeq": ["error", "always"],
      "no-var": "error",
      "prefer-const": "warn",
      "curly": ["error", "all"],
    },
  },
  {
    files: ["**/*.test.js"],
    rules: {
      "no-console": "off",
    },
  },
];

Cada regra aceita tres niveis: "off" (desligada), "warn" (aviso) e "error" (bloqueia). Algumas regras aceitam opcoes extras no formato array, como ["error", "always"].

A secao files permite aplicar regras diferentes para arquivos especificos. No exemplo acima, arquivos de teste podem usar console.log livremente -- faz sentido para debugging durante o desenvolvimento de testes.

Criando regras customizadas

As vezes, as regras prontas nao cobrem uma necessidade especifica do seu projeto. Digamos que voce quer proibir chamadas HTTP sem tratamento de erro. Voce pode criar um plugin ESLint customizado:

// eslint-plugin-api-safety/rules/require-error-handling.js
export default {
  meta: {
    type: "problem",
    docs: {
      description: "Exige try/catch em chamadas fetch",
    },
    messages: {
      missingErrorHandling: "Chamadas fetch devem ter tratamento de erro.",
    },
  },
  create(context) {
    return {
      CallExpression(node) {
        if (node.callee.name === "fetch") {
          const parent = node.parent;
          // Verifica se o fetch esta dentro de try/catch
          let current = parent;
          while (current) {
            if (current.type === "TryStatement") return;
            current = current.parent;
          }
          context.report({ node, messageId: "missingErrorHandling" });
        }
      },
    };
  },
};

Essa regra e especialmente util quando seu projeto consome APIs externas. Se voce integra com servicos de consulta de CPF ou CNPJ, um fetch sem try/catch pode derrubar toda a experiencia do usuario por causa de um timeout nao tratado. Regras de lint customizadas protegem contra esses descuidos.

Presets e configuracoes compartilhadas

Ninguem precisa configurar 200 checagens do zero. Use presets:

  • eslint:recommended -- regras basicas que pegam bugs obvios
  • airbnb -- o guia de estilo mais popular, opinado e rigoroso
  • standard -- sem ponto e virgula, popular em projetos open source
  • google -- estilo do Google, equilibrado

Para times, crie um pacote npm com suas regras de lint e compartilhe entre projetos. Isso garante consistencia em toda a organizacao. Um eslint-config-minha-empresa evita que cada repositorio invente suas proprias convencoes.

// eslint-config-minha-empresa/index.js
import js from "@eslint/js";

export default [
  js.configs.recommended,
  {
    rules: {
      "no-console": "warn",
      "no-debugger": "error",
      "prefer-const": "error",
      "no-unused-vars": ["error", { argsIgnorePattern: "^_" }],
      "max-lines-per-function": ["warn", { max: 50 }],
    },
  },
];

O padrao argsIgnorePattern: "^_" permite variaveis prefixadas com underscore sem erro -- uma convencao comum para parametros obrigatorios que voce nao usa, tipo callbacks onde so o segundo argumento importa.

Se voce desenvolve APIs profissionais, manter regras de lint padronizadas entre todos os servicos reduz o atrito em code review. O Hub do Desenvolvedor, por exemplo, oferece APIs de consulta de CPF, CNPJ e CEP que seguem padroes consistentes -- e a qualidade do codigo por tras disso passa por processos como esse.

🔍 INSIGHT: Times que publicam suas configuracoes de linting como pacotes compartilhados reduzem em media 40% o tempo de onboarding de novos devs. O linter ensina as convencoes do projeto automaticamente, sem precisar de um documento de 50 paginas que ninguem le.

Regras de lint que todo projeto precisa

Regras de deteccao de bugs

Estas sao as checagens que realmente salvam seu codigo de bugs em producao. Nao sao questao de estilo, sao questao de corretude:

  • no-undef -- impede uso de variaveis nao declaradas. Pega typos como cosnt em vez de const.
  • no-unreachable -- detecta codigo depois de return, throw ou break que nunca sera executado.
  • no-dupe-keys -- impede chaves duplicadas em objetos, onde a segunda sobrescreve a primeira silenciosamente.
  • no-async-promise-executor -- barra new Promise(async (resolve) => ...), um padrao que esconde erros.
  • no-loss-of-precision -- detecta numeros literais que perdem precisao em JavaScript, tipo 9007199254740993 que vira 9007199254740992.

Em Python com Ruff, as equivalentes sao:

  • F841 -- variavel local atribuida mas nunca usada
  • E711 -- comparacao com None usando == em vez de is
  • B006 -- argumento default mutavel em funcao (aquele bug classico da lista vazia)
  • S105 -- senhas hardcoded no codigo

Regras de seguranca

Regras de lint focadas em seguranca sao as mais subestimadas. Elas pegam vulnerabilidades antes que qualquer scanner de seguranca entre em acao:

Regra Ferramenta O que detecta
no-eval ESLint Uso de eval() que permite injecao de codigo
detect-non-literal-regexp eslint-plugin-security RegExp com input do usuario (ReDoS)
S608 Ruff/Bandit SQL injection via string formatting
S301 Ruff/Bandit Uso de pickle (desserializacao insegura)
react/no-danger eslint-plugin-react dangerouslySetInnerHTML (XSS)
no-secrets/no-secrets eslint-plugin-no-secrets Strings que parecem tokens/chaves de API

A regra no-secrets merece atencao. Ela usa entropia de Shannon -- uma medida matematica de aleatoriedade de uma string -- para detectar strings que parecem chaves de API ou tokens. Se voce trabalha com integracao de APIs -- chamando endpoints de consulta de CPF ou CNPJ, por exemplo -- essa regra ajuda a garantir que credenciais nao vazem no codigo commitado.

Regras de qualidade e manutencao

Alem de bugs e seguranca, existem regras de lint que protegem a saude do codigo ao longo do tempo:

  • complexity -- limita a complexidade ciclomatica de funcoes. Um if dentro de outro if dentro de um for vira espaguete rapido. Mantenha abaixo de 10.
  • max-depth -- limita aninhamento de blocos. Se voce tem 5 niveis de indentacao, seu codigo precisa de refatoracao.
  • max-lines-per-function -- funcoes longas sao dificeis de testar e entender. 50 linhas e um limite saudavel.
  • no-magic-numbers -- numeros soltos no codigo sem explicacao viram misterio em 3 meses. Use constantes nomeadas.
  • prefer-template -- template literals sao mais legiveis que concatenacao com +.
// Ruim -- magic number, concatenacao
if (response.status === 429) {
  console.log("Erro na API: " + endpoint + " retornou limite excedido");
}

// Bom -- constante nomeada, template literal
const HTTP_TOO_MANY_REQUESTS = 429;
if (response.status === HTTP_TOO_MANY_REQUESTS) {
  console.log(`Rate limit atingido no endpoint ${endpoint}`);
}

Esse tipo de regra nao parece urgente, mas projetos com 2+ anos de vida agradecem. Codigo limpo hoje e menos divida tecnica amanha.

Integracoes: lint no editor, CI/CD e code review

Lint no editor de codigo

A melhor hora para corrigir um problema de lint e enquanto voce digita. Configurar o linter no editor cria um feedback loop instantaneo.

VS Code:

Instale a extensao ESLint (mais de 35 milhoes de instalacoes). Adicione no settings.json:

{
  "editor.codeActionsOnSave": {
    "source.fixAll.eslint": "explicit"
  },
  "eslint.validate": [
    "javascript",
    "typescript",
    "javascriptreact",
    "typescriptreact"
  ]
}

Ao salvar o arquivo, o ESLint corrige automaticamente tudo que for auto-fixavel -- indentacao, aspas, imports ordenados. Voce nem precisa pensar nisso.

Neovim:

Use o nvim-lint ou configure o LSP nativo com eslint-lsp. A integracao e tao boa quanto no VS Code, com diagnosticos inline e fix automatico.

JetBrains (WebStorm/IntelliJ):

O ESLint vem integrado nativamente. Va em Settings > Languages & Frameworks > JavaScript > Code Quality Tools > ESLint e selecione "Automatic ESLint configuration".

Lint no pipeline de CI/CD

Rodar o linter no CI garante que nada passe sem revisao, mesmo que o dev esqueca de configurar o editor. Um workflow basico para GitHub Actions:

name: Lint
on: [push, pull_request]

jobs:
  lint:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
          cache: npm
      - run: npm ci
      - run: npx eslint . --max-warnings 0

O flag --max-warnings 0 faz diferenca. Sem ele, warnings passam silenciosamente e se acumulam. Tratar warnings como erros no CI forca o time a resolver tudo antes do merge.

Para projetos Python:

      - run: pip install ruff
      - run: ruff check . --output-format=github

O formato github gera anotacoes diretamente nos arquivos do pull request, o que facilita a visualizacao dos problemas.

Lint no pre-commit com Husky

O Husky permite rodar scripts antes de cada commit. Combinado com o lint-staged, ele roda o linter apenas nos arquivos que voce modificou -- rapido e cirurgico:

// package.json
{
  "lint-staged": {
    "*.{js,ts,jsx,tsx}": ["eslint --fix", "prettier --write"],
    "*.{css,scss}": ["stylelint --fix"],
    "*.py": ["ruff check --fix", "ruff format"]
  }
}

npx husky init
echo "npx lint-staged" > .husky/pre-commit

Esse setup pega problemas antes de poluir o historico de commits. Nada de "fix lint" como mensagem de commit -- o codigo ja sai limpo do seu terminal.

Lint em code review automatizado

Ferramentas como o Danger.js e o Reviewbot integram resultados de lint diretamente nos pull requests. Em vez de um dev senior apontar "voce esqueceu de tratar essa promise", o bot comenta automaticamente.

Para times que constroem integracoes com APIs, manter lint automatizado no code review e fundamental. Ele garante que cada chamada tenha tratamento de erro, tipagem correta e nenhum segredo exposto no codigo. APIs como as do Hub do Desenvolvedor exigem esse nivel de cuidado.

📈 RESULTADO: Estudos de engenharia de software indicam que equipes com linting automatizado no CI reduzem bugs em producao em 15-20% e gastam significativamente menos tempo em revisao de codigo. O investimento de 2 horas configurando lint economiza centenas de horas ao longo do projeto.

Monitorando a saude do lint ao longo do tempo

Nao basta configurar e esquecer. Use o SonarQube ou o Code Climate para acompanhar metricas de qualidade ao longo do tempo. Eles mostram se a divida tecnica esta aumentando ou diminuindo, quantas violacoes de lint existem por sprint e quais regras o time mais viola.

Uma metrica util e o "lint fix rate" -- a porcentagem de problemas de lint corrigidos em relacao aos introduzidos. Se esse numero esta abaixo de 100%, voce esta acumulando debito. Acima de 100%, esta melhorando progressivamente.

Erros comuns e boas praticas com regras de lint

Os 5 erros que times cometem com lint

1. Desativar regras em vez de corrigir o codigo

Todo dev ja colocou um // eslint-disable-next-line porque "nao dava tempo de arrumar". Uma ou duas vezes, ok. Mas quando voce tem 200 disables espalhados pelo projeto, o linter virou decoracao.

Crie uma regra organizacional: todo disable precisa de um comentario explicando o motivo. Se nao tem justificativa, nao desativa.

2. Configurar regras demais de uma vez

Um projeto legado com 100k linhas de codigo vai gerar milhares de erros se voce ativar o preset mais rigoroso. O time desanima, desativa tudo e volta a estaca zero.

Comece com eslint:recommended. Adicione 5 regras por sprint. Em 3 meses voce tem um setup robusto sem trauma.

3. Nao diferenciar warnings de errors

Se tudo e error, o dev fica dessensibilizado. Se tudo e warning, ninguem corrige. Use error para coisas que causam bugs reais (como no-undef). Use warning para melhorias de qualidade (como prefer-const). E no CI, trate warnings como blockers para nao acumular.

4. Ignorar lint em arquivos de teste

Testes sao codigo tambem. Testes mal escritos quebram, dao falsos positivos e ninguem confia neles. Adapte as regras -- permita no-console e max-lines-per-function mais frouxo -- mas nao desative o lint inteiro.

5. Nao alinhar formatador com linter

Prettier e ESLint brigando e o inferno. O Prettier formata, o ESLint reclama da formatacao, voce corrige, o Prettier reformata. Configure eslint-config-prettier para desativar regras de estilo do ESLint que conflitam com o Prettier. Problema resolvido.

Quando faz sentido desativar uma regra

Existem cenarios legitimos para desativar regras de lint. Nao e pecado, e pragmatismo:

  • Codigo gerado automaticamente -- arquivos de migracao de banco, tipos gerados por GraphQL codegen ou OpenAPI. Coloque esses diretorios no .eslintignore.
  • Prototipagem rapida -- durante um hackathon ou prova de conceito, lint rigoroso atrapalha a velocidade. Mas ative antes de ir para producao.
  • Conflito com padrao da linguagem -- algumas regras do ESLint nao fazem sentido em TypeScript porque o compilador ja cobre. O typescript-eslint resolve a maioria desses conflitos.
  • Bibliotecas de terceiros -- se voce usa uma API que retorna dados num formato estranho, desative camelcase naquele arquivo especifico.

Desativar com intencao e documentacao e diferente de desativar por preguica.

Para onde o linting esta indo

Ferramentas como o Biome e o Ruff mostram que performance importa -- ninguem quer esperar 30 segundos para o linter rodar. A tendencia e combinar linting, formatacao e type-checking numa unica ferramenta rapida.

Outro movimento interessante: o uso de IA para sugerir correcoes de lint. O GitHub Copilot ja sugere fixes inline, e ferramentas como o Amazon CodeGuru fazem revisao automatizada com regras que vao alem do lint tradicional, detectando memory leaks, race conditions e padroes ineficientes.

Para quem trabalha com APIs, manter o linting atualizado garante que seu codigo de integracao permaneca robusto. Se voce precisa de APIs confiaveis para consulta de CPF, CNPJ e CEP, o Hub do Desenvolvedor oferece endpoints bem documentados e com SDKs que ja seguem boas praticas de lint.

Regras de lint sao a base da qualidade de codigo em qualquer projeto profissional. Do dev solo que trabalha no proprio SaaS ate times de 200 pessoas, o linting automatizado previne bugs, padroniza estilo e economiza tempo real de desenvolvimento.

O caminho e simples: escolha a ferramenta certa para sua linguagem, comece com regras conservadoras, integre no editor e no CI, e evolua gradualmente. Em poucas semanas voce vai sentir a diferenca na qualidade do codigo e na velocidade das revisoes.
